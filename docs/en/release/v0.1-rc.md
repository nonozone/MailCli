# MailCLI v0.1 RC Release Notes

## Summary

`v0.1 RC` marks the first release candidate where MailCLI is credible as an open, AI-native email interface rather than just a parser experiment.

The project now has a usable end-to-end loop for:

- parsing local email into `StandardMessage`
- listing and fetching inbox messages from IMAP accounts
- syncing recent messages into a local index and searching them locally
- inspecting thread summaries and full local threads for agent triage
- compiling and sending outbound drafts and replies
- integrating with Python, shell, and external agent providers

## What Is In Scope

- parser-first inbound normalization
- baseline IMAP read path
- local sync, search, and thread retrieval
- SMTP-backed send path for IMAP-style accounts
- stable machine-facing JSON contracts
- agent examples and provider handoff examples
- contributor-facing extension docs

## Key Additions Before RC

- `parse`, `list`, `get`, `send`, and `reply` command set
- `sync`, `search`, `threads`, and `thread` local retrieval loop
- `DraftMessage`, `ReplyDraft`, and `SendResult`
- reply threading support through `reply_to_message_id` and `reply_to_id`
- `multipart/alternative` outbound composition
- attachment packaging in the composer
- env-backed config secrets for password fields
- richer inbox list summaries with mailbox/limit controls
- clearer sync/index output counts and richer thread triage fields
- stronger HTML body extraction and tracked URL normalization heuristics
- external provider contract and OpenAI provider example
- expanded parser fixture corpus

## Stable Integration Boundary

The intended stable boundary for `v0.1 RC` is:

- `mailcli parse`
- `mailcli list`
- `mailcli get`
- `mailcli sync`
- `mailcli search`
- `mailcli threads`
- `mailcli thread`
- `mailcli send`
- `mailcli reply`
- `StandardMessage`
- `DraftMessage`
- `ReplyDraft`
- `SendResult`

## Still Heuristic

The following are useful today, but should still be treated as evolving:

- action extraction coverage
- verification-code extraction outside common layouts
- complex HTML cleanup and body selection for unusual templates
- token estimates

## Explicit Non-Goals For This RC

- full terminal mail client behavior
- OAuth-heavy auth integrations in core
- runtime plugin loading
- SQLite/FTS-backed local storage
- provider-specific business rules in shared parser/composer layers

## Recommended Next Step After RC

- stronger parser corpus
- parser contributor onboarding and driver harnesses
- richer local search semantics and ranking
- more provider implementations after the shared contracts settle
